# 性能故事：矢量化与 Rust 执行路径

当我们谈 LTSeq 的性能时，重点不是“它更快”，而是“它为什么在某些场景里更稳定地快”。LTSeq 的性能故事并不神秘，它来自两个明确选择：**矢量化的数据模型**与**Rust 内核**。这两件事让 LTSeq 可以在保持语义清晰的同时，尽量让性能可预测。

这一篇的核心问题是：**LTSeq 的性能优势来自哪里，边界又在哪里？**

## 1. 心智模型：性能来自执行路径，而不是语法

你写的是 Python，但执行不一定发生在 Python。LTSeq 把“表达”和“执行”分开：

- Python 负责表达（lambda → 表达式树）
- Rust 负责执行（表达式 → 计划 → 算子）

真正决定性能的是后半段：执行路径是否矢量化、是否避免逐行 Python 逻辑。

## 2. 矢量化的核心：批处理而非逐行

LTSeq 的 Rust 内核会尽量以“批次”处理数据，这意味着：

- 一次处理一块数据（RecordBatch）
- 避免 Python 层逐行循环
- 利用 CPU cache 的局部性

这就是所谓的矢量化路径。它不是魔法，而是计算模型的差异：**逐行是解释型，批处理是编译型**。

## 3. Rust 内核的意义：稳定与可控

Python 的优势是表达力，但在执行层面它有天然的开销。LTSeq 把执行交给 Rust 内核，是为了获得：

- 更稳定的性能
- 更可预测的内存行为
- 更容易做底层优化

在 LTSeq 中，`_inner` 就是 Python 与 Rust 的边界。只要表达式能被捕获并序列化，执行就有机会走到 Rust 内核。

## 4. 性能来自“快路径”的覆盖范围

LTSeq 的性能并不是“对所有操作都快”，而是“对走在快路径上的操作快”。

快路径通常意味着：

- 表达式可序列化
- 排序状态清晰
- 使用了内核支持的算子

当这些条件满足时，LTSeq 可以稳定地走向 Rust + 矢量化路径。否则就会进入 fallback，性能回落到 pandas 或 Python。

## 5. 性能边界在哪里

LTSeq 的性能边界主要来自三个方面：

- **表达式边界**：复杂 Python 逻辑无法被捕获
- **算子边界**：某些操作在 Rust 内核尚未实现
- **数据边界**：极端的数据倾斜或无序会削弱优化效果

这也是为什么 LTSeq 强调“明确排序”和“表达式约束”，因为这些约束直接决定了性能路径是否可用。

## 6. 为什么性能必须与语义绑定

很多系统追求“更快”，但速度如果建立在语义模糊上，最终会造成结果不可解释。LTSeq 的路线是：

- 先保证语义清晰
- 在清晰语义的基础上优化

这让它的性能优势更像“稳定收益”，而不是“偶然加速”。

## 7. 工程权衡：Rust 依赖与生态兼容

性能的代价也很明确：

- **Rust 依赖增加**：需要编译与绑定
- **部署复杂度上升**：环境要求更高
- **生态兼容挑战**：与纯 Python 工具链存在差异

LTSeq 用这些代价换来的是“执行稳定性”。这是一种工程上的取舍，而不是纯技术的炫技。

## 8. 价值总结：性能是结果，不是目的

LTSeq 的性能故事可以总结为一句话：**性能是正确语义与可控执行路径的结果**。

它不会让所有事情都变快，但当你在序列语义内表达问题时，它能稳定地提供更好的执行路径。这种“可预测的快”，比“偶尔的快”更重要。

## 9. 系列小结

至此，我们完成了整个 LTSeq 系列的核心解剖：

- 从“序列优先”的问题域出发
- 深入表达式捕获、排序契约、窗口与连续分组
- 理解惰性关联、lookup 重写、join 家族
- 看到 partition 与 grouping 的语义差异
- 最后回到执行层，理解性能的来源与边界

LTSeq 的故事不是“功能堆叠”，而是“语义选择”。它用一套明确的工程取舍，换来了更适合序列分析的表达与执行模型。

如果你处理的世界是“序列”，LTSeq 提供的不是一把更锋利的刀，而是一整套更合适的手术工具。
