# Lambda 如何变成执行计划：LTSeq 的表达式捕获管线

当你在 LTSeq 里写下 `lambda r: r.price > 100`，你以为自己在执行 Python，但 LTSeq 做的其实是“记录你想做什么”。这不是小聪明，而是整个引擎能被优化、被跨语言执行的前提。理解这条“表达式捕获管线”，你就能理解 LTSeq 为什么能把序列操作做得更稳、更可控。

这一篇的核心问题是：**一段 lambda 是如何从“Python 语句”变成“可执行计划”的？**

## 1. 问题背景：为什么要把 lambda 变成计划

如果系统直接执行 Python lambda，就会有两个现实问题：

- **无法优化**：Python 函数就是黑盒，执行引擎不知道里面发生了什么。
- **无法跨语言**：Rust 内核无法理解你的 Python 逻辑，只能在 Python 层慢慢跑。

LTSeq 的目标不是“让 Python 做得更快”，而是**让 Python 负责表达，让 Rust 负责执行**。所以它必须把 lambda 变成一种“可序列化的表达”，才能交给下层 Rust 内核。

## 2. 心智模型：你写的是“描述”，不是“执行”

可以这样理解：

- 你写 lambda，就像你在画一张“执行草图”。
- LTSeq 不会立即执行它，而是把这张草图保存下来。
- 最终由执行引擎根据草图生成真正的执行路径。

这和 SQL 的本质很像：SQL 不是执行代码，而是描述你要的结果。LTSeq 也走的是“描述式”路线，只不过它用的是 Python 语法来写。

## 3. 表达式捕获的三个关键角色

在源码里，这条管线由三个角色共同完成：

### 3.1 SchemaProxy：让 `r.col` 变成表达式

`SchemaProxy`（`py-ltseq/ltseq/expr/proxy.py`）是你在 lambda 里看到的 `r`。它并不代表真实的数据行，而是一个“行的代理”。

当你写 `r.price` 时，`SchemaProxy.__getattr__()` 不会返回真实值，而是返回一个 `ColumnExpr("price")`。也就是说，**属性访问本身就被捕获成了表达式节点**。

这一步完成了“从语法到表达式”的第一跳。

### 3.2 Expr 体系：把运算构成一棵树

`ColumnExpr` 只是表达式树的叶子节点。真正的“树”来自运算的组合：

- `r.price > 100` 会变成 `BinOpExpr`（二元操作表达式）
- `r.name.s.contains("A")` 会变成 `CallExpr`（函数/方法调用表达式）

这些表达式类型定义在 `py-ltseq/ltseq/expr/types.py` 和相关模块里。它们最重要的能力是：**能被序列化**。

最终，每个表达式都有一个 `serialize()` 方法，输出一个结构化的 dict。例如：

- `ColumnExpr("price")` → `{ "type": "Column", "name": "price" }`
- `r.price > 100` → `{ "type": "BinOp", "op": "Gt", "left": ..., "right": ... }`

这一步完成了“从表达式到可序列化结构”的第二跳。

### 3.3 LTSeq._capture_expr：把树交给内核

当你调用 `filter()`、`derive()`、`agg()` 等 API 时，LTSeq 会调用内部的 `_capture_expr()`（定义在 `py-ltseq/ltseq/core.py`）来执行这条捕获链。

该函数做的事情很直接：

- 用 `SchemaProxy` 执行你的 lambda，得到表达式对象
- 调用 `serialize()` 把表达式转成 dict
- 把 dict 交给 `_inner`（Rust 内核）

这一步完成了“从可序列化结构到执行计划入口”的第三跳。

## 4. 一个简单例子的“白盒解释”

你写：

- `t.filter(lambda r: r.price > 100)`

实际发生的过程是：

1) `r` 不是真实数据，而是 `SchemaProxy`
2) `r.price` → `ColumnExpr("price")`
3) `ColumnExpr("price") > 100` → `BinOpExpr`
4) `BinOpExpr.serialize()` → dict 结构
5) dict 传给 Rust 内核，生成执行路径

注意，这里没有一步是在执行 Python 的“过滤”。Python 只负责构造表达式，真正的执行由 Rust 完成。

## 5. 这条管线带来的工程价值

### 5.1 可优化
表达式树可见，执行引擎可以优化它，比如：

- 合并过滤条件
- 选择更高效的执行算子
- 在可能时走向矢量化路径

### 5.2 可扩展
表达式体系清晰后，你可以新增表达式类型（例如新的窗口函数），而不需要改动外部 API。也就是说，**扩展发生在表达式层，而不是用户层**。

### 5.3 可跨语言
因为表达式序列化为结构化 dict，它不依赖 Python 运行时。这使得 Rust 内核可以独立实现执行逻辑，并且保持性能优势。

## 6. 这条管线的代价与边界

表达式捕获不是免费的，它要求你遵守一些约束：

### 6.1 lambda 需要“可序列化”
如果你写了复杂的 Python 逻辑，比如循环、条件分支、调用外部函数，LTSeq 不一定能捕获到表达式。这不是缺陷，而是架构边界。

### 6.2 表达式是“受限语法”
LTSeq 只支持它能识别的表达式类型。例如：

- 列访问
- 基本算术
- 预定义函数或访问器（如 `.s`、`.dt`）

超出这些范围，可能会走 fallback 路径，或直接报错。

### 6.3 学习成本
你需要理解“表达式不是执行”的概念。对于习惯“写 Python 就是执行”的用户，这是一个认知转变。

## 7. 和 SQL 的类比：为何更接近“声明式”

LTSeq 的表达式捕获让它更像 SQL：

- SQL 是声明式：你描述想要的结果
- LTSeq 的 lambda 也是声明式：你描述想要的计算

唯一不同的是，LTSeq 用 Python 语法来表达声明式结构。这让它既保留了 Python 的亲和力，又获得了声明式系统的可优化性。
