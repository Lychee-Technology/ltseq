# 集合操作：union / intersect / diff 的数据策略

在序列分析里，集合操作看似“传统”，但它们承担的是非常基础的责任：**维护 schema 一致性与结果可解释性**。你可以把 `union`、`intersect`、`diff` 理解为“数据契约的校验器”。它们之所以被 LTSeq 放在核心 API 中，是因为序列分析最终仍然会回到集合层面的组合与对比。

这一篇聚焦 LTSeq 的集合操作：这些操作如何保持语义一致，以及它们背后的工程权衡。

## 1. 痛点：集合操作容易破坏数据契约

在很多工具里，集合操作只关心“能不能合并”，不关心“合并后是否可靠”。但在真实工程中，集合操作常常是数据质量问题的放大器：

- 两张表 schema 不一致，却被硬拼在一起
- 列顺序不同导致结果错位
- 数据类型不兼容导致隐式转换

LTSeq 的集合操作设计强调：**先保证 schema 一致，再谈结果**。

## 2. 心智模型：集合操作是“契约检查”

可以把集合操作理解成“契约检查器”：

- `union`：只有当 schema 完全一致，才能拼接
- `intersect`：只有可对齐的列才有意义
- `diff`：必须保证“比较对象”是同一结构

这让集合操作不再只是“数据拼接”，而是**一种结构保证**。

## 3. 源码机制：SetOpsMixin 的直达路径

LTSeq 的集合操作集中在 `SetOpsMixin`（见 `py-ltseq/ltseq/advanced_ops.py`）。这些操作的特点是：

- Python 层做参数检查（是否是 LTSeq、schema 是否初始化）
- 合法后直接交给 `_inner`（Rust 内核）执行

这种设计很直接：集合操作不是“表达式计算”，而是“结构级组合”，因此不需要复杂的表达式捕获管线。

## 4. 三类操作的语义差别

### 4.1 union：垂直拼接
`union` 在 LTSeq 中对应的是 UNION ALL，语义是“把两张表纵向拼起来”。它的核心前提是：**schema 必须一致**。这保证了拼接结果不会出现列错位或隐式转换。

### 4.2 intersect：交集
`intersect` 用来找两张表的交集，可以指定 `on` 作为 key。它强调的是“哪些记录同时存在”。这在对比数据版本、找共同事件时很有用。

### 4.3 diff：差集
`diff` 返回“左表有而右表没有”的部分。它常用于增量对比或变化检测，同样可以指定 `on` 作为 key。

这三类操作看似简单，但共同点是：它们都需要非常明确的结构契约，否则结果无法解释。

## 5. 工程权衡：严格性 vs 灵活性

LTSeq 的集合操作更偏“严格”。这带来一些权衡：

- **严格性提升**：避免隐式错误，结果更可靠
- **灵活性降低**：不能像某些工具那样自动对齐列或隐式转换

这种设计风格和 LTSeq 的整体策略一致：宁可让用户明确处理不一致，也不让系统暗中猜测。

## 6. 为什么集合操作依然重要

序列分析最终会走向“版本对比”和“数据合并”：

- 你可能需要把多个时间段的结果拼接起来
- 你可能需要对比两次计算结果的差异
- 你可能需要找出“新增事件”或“消失事件”

这些都依赖集合操作。LTSeq 通过严格的 schema 契约，确保这些操作不会悄悄损坏语义。
