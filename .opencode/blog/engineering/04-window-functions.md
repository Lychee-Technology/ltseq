# 窗口函数的执行路径：shift/rolling/diff 的底层逻辑

在序列数据里，窗口函数并不是“高级技巧”，而是日常问题的答案。你想知道上一条记录、过去 7 天的滚动均值、累计变化，这些本质上都是“窗口”。LTSeq 之所以强调排序与表达式捕获，就是为了让窗口函数能稳定、可控、且高效地执行。

这一篇我们从产品视角进入源码视角：**窗口函数在 LTSeq 里是如何被识别并走向执行路径的？**

## 1. 为什么窗口函数是序列分析的核心

窗口函数处理的是“相邻关系”，而不是“全局统计”。它解决的问题包括：

- 上一个值与当前值的差（`diff`）
- 固定区间内的滚动均值（`rolling`）
- 时间序列的位移对齐（`shift`）

这些需求在交易、日志、行为序列中极其常见。如果窗口函数无法稳定运行，序列分析几乎就失去了意义。

## 2. 心智模型：窗口就是“滑动的观察窗”

想象你站在一条传送带旁边，手里有一个固定大小的观察框。传送带上的物品流过时，你每次只看框里的那一段。这就是窗口函数：

- `shift` 是向前或向后看一格
- `rolling` 是看一段连续区间
- `diff` 是比较“当前”和“上一格”的差异

这个模型帮助你理解：**窗口操作不是全局聚合，而是序列上的局部视角**。

## 3. 执行路径的前提：排序是入口条件

LTSeq 要求窗口函数之前显式排序，并不是形式主义，而是为了保证“观察窗的移动方向一致”。没有排序，窗口就不知道该沿着哪条轴滑动。

在内部，排序状态由 `_sort_keys` 追踪。这意味着：

- 当排序存在时，窗口函数可以走向稳定执行路径
- 当排序缺失时，系统会拒绝或提示，避免生成不可靠结果

## 4. 窗口函数如何被识别

LTSeq 的窗口能力依赖表达式捕获机制：你写的 `r.col.shift(1)` 并不会立即执行，而是被转换成 `CallExpr`，并最终序列化为 dict 结构（见 `py-ltseq/ltseq/expr/types.py`）。

这意味着系统可以在“表达式层”识别窗口函数：

- 如果表达式里出现 `shift`、`rolling` 等关键调用
- 系统就能判断这是一个窗口计算

一旦识别成功，LTSeq 会将该表达式交给 Rust 内核执行，而不是在 Python 层循环计算。

## 5. 走向 Rust 内核的执行路径

表达式捕获的最终目标是把计算交给 `_inner`。在 LTSeq 里，窗口函数通常遵循这样的路径：

1) Python 层捕获表达式
2) 表达式序列化为 dict
3) 交给 Rust 内核
4) 内核选择矢量化/批处理方式执行

这种路径有两层价值：

- **性能**：Rust 内核能以矢量化方式处理窗口，不必逐行 Python 循环
- **一致性**：表达式树可被统一调度，避免不同功能之间的语义偏差

## 6. 窗口不是“额外功能”，而是核心管道的一部分

在 LTSeq 的设计里，窗口函数并不是在“外层加一个特例”，而是融入了表达式体系：

- `shift()` 只是 `CallExpr` 的一种
- `rolling().mean()` 只是表达式链的组合
- 这些表达式在序列化后与其他表达式同级处理

这让窗口函数可以与 `filter`、`derive`、`agg` 等操作统一组合，而不是成为难以维护的“特殊规则”。

## 7. 工程权衡：为什么窗口函数看似简单却很“硬”

窗口函数的价值来自序列语义，但也带来一些明确的权衡：

- **排序依赖**：窗口必须建立在明确顺序上，缺少排序会被拒绝
- **表达式约束**：窗口表达式必须被捕获为可序列化结构
- **边界行为**：窗口的起止边界需要一致定义，否则会引发统计偏差

这些限制不是“功能缺失”，而是序列系统必须保证的稳定性。

## 8. 为什么这条路径很重要

窗口函数看似只是 API 的一个选项，但它在工程上有三层意义：

- **语义一致性**：所有窗口操作都建立在排序契约之上
- **性能稳定性**：避免 Python 层逐行计算
- **表达统一性**：窗口只是表达式树的一部分，不需要额外逻辑分支

换句话说，窗口函数不是附属功能，而是 LTSeq 能“真正序列化分析”的核心体现。
