# Lookup：把列级增强偷偷变成 Join

很多业务分析都有“查字典”的需求：订单表里只有 `product_id`，你想补上 `product_name`；日志里只有 `user_id`，你想补上用户画像。传统做法是 join，一次性把整张维表拉进来。但在真实工程里，你经常只需要一两个字段，完整 join 反而是负担。

LTSeq 的 `lookup` 正是在这个痛点上做了一个工程化设计：**它让“列级增强”看起来像函数调用，但在底层会被重写成 join**。这既保留了语法简洁，也让执行引擎保持一致。

这一篇讨论的核心问题是：**lookup 为什么不是魔法，而是一种受控的 join 重写？**

## 1. 痛点：轻量需求被迫使用重型 Join

如果你只想补一列信息，但只能通过 join 来做，你会面临几个问题：

- join 会把整张维表带进来，产生无用列
- join 的列名冲突需要额外处理
- 在大表场景下，join 成本高且难以推迟

而实际需求可能只是“按 id 查个名字”。lookup 要解决的正是这种“轻量增强”场景。

## 2. 心智模型：lookup 像字典查表

你可以把 `lookup` 理解成“字典查表”：

- 输入是某列的键
- 输出是维表里的对应值

这种语义比 join 更接近业务语言，也更容易阅读。但为了保持执行一致性，LTSeq 在底层并不是“真的在 Python 里查字典”，而是把它重写成 join 计划。

## 3. 源码机制：LookupMixin 的重写流程

`lookup` 的核心逻辑在 `py-ltseq/ltseq/lookup.py` 中，由 `LookupMixin` 提供。它的流程大致是：

1) 在 `derive()` 中捕获表达式
2) 检测表达式树里是否包含 `Lookup`
3) 提取 lookup 的目标表、目标列、join key
4) 对当前表执行 join
5) 重写表达式，让它引用 join 后的列

这意味着：**lookup 只是一个“语法入口”，真正执行路径仍然是 join**。

## 4. 为什么要重写，而不是直接查表

原因主要有三点：

- **执行一致性**：统一走 join 路径，便于优化与调度
- **可扩展性**：lookup 的输出列依然是正常列，后续操作不需要特殊处理
- **性能可控**：Rust 内核可以选择合适的 join 算法

如果 lookup 真的是逐行查表，性能会高度依赖 Python 运行时，且难以优化。

## 5. 设计细节：alias 与列命名

lookup 重写时会引入 alias 前缀（例如 `_lkp_products`），避免字段冲突。目标列会变成：

- `{alias}_{column}`

然后表达式会被替换成对该列的直接引用。这样做的好处是：

- join 结果能与普通列一样被使用
- schema 一致且可追踪

## 6. 工程权衡：语法简洁 vs 执行透明

lookup 的设计选择了一种折中：

- **对用户简洁**：你写的是“查找”语义
- **对引擎透明**：系统仍旧走 join 执行路径

这带来的权衡是：

- 用户可能不了解背后的 join 成本
- alias 前缀需要理解，否则列名会显得陌生
- lookup 的能力边界由 join 能力决定

这也是为什么 LTSeq 在文档中强调：lookup 是语法糖，但不是免费的。

## 7. 价值总结：让增强更接近业务表达

lookup 的最大价值不是性能，而是**表达方式更接近业务语言**：

- “按 id 查名字”比“join 表”更符合业务思考方式
- 结果仍然是标准列，后续计算无缝衔接

这让 LTSeq 在“语义友好”和“执行一致”之间找到一个很好的平衡点。
