# 连续分组：group_ordered 与 group_sorted 的真实语义

如果你习惯了传统 `groupby`，你会把“分组”理解为“把相同值聚在一起”。但在序列数据里，真正重要的往往不是“相同值”，而是**相同值的连续段**。这正是 LTSeq 的 `group_ordered` 和 `group_sorted` 要解决的问题。

这一篇聚焦一个被很多工具忽略的能力：**连续分组（ordered grouping）**。它不是语法糖，而是序列分析的核心语义。

## 1. 痛点：传统 groupby 无法表达“连续段”

考虑一个真实场景：你想识别“连续上涨区间”。如果用传统 groupby，你会发现它只关心“值相同”，而不关心“是否连续”。结果是：

- 所有上涨行被聚在一起
- 连续与不连续的区间被混在一组

但在序列分析里，**连续段本身才是信息**。一个上涨段与另一个上涨段之间的断点，往往比上涨本身更重要。

LTSeq 提供的连续分组，正是为了把“连续段”变成一等能力。

## 2. 心智模型：把“连续段”当作状态片段

可以把序列想象成一条时间轴。每当状态发生变化，时间轴就被切成一个新的片段。连续分组就是在做这件事：

- 相同值并不一定属于同一个片段
- 只有“连续相同”才会被归为同一组

这种模型非常适合：

- 状态机的运行区间
- 行为连续段（活跃/沉默）
- 价格或指标的趋势段

## 3. 两个 API 的语义差别

LTSeq 提供两个连续分组 API：

- `group_ordered`：按现有顺序分组，只识别连续相同值
- `group_sorted`：假设数据已经按 key 全局排序，因此连续相同值等同于全局相同值

这两个 API 的底层机制相同，但语义不同：

- `group_ordered` 更贴近“原始序列”
- `group_sorted` 强调“排序后的序列”，更接近传统分组但仍保留连续语义

这个区分让 LTSeq 可以在“序列语义”和“传统分组”之间提供更精确的表达方式。

## 4. 源码机制：NestedTable 与 group_id

连续分组在 LTSeq 中由 `NestedTable` 实现（见 `py-ltseq/ltseq/grouping/nested_table.py`）。它的核心机制是：

1) 计算每一行的分组 key
2) 如果 key 与前一行不同，就生成新的 `__group_id__`
3) `__group_id__` 标识连续段，而不是全局分组

这样得到的分组结构与传统 groupby 不同，它强调“段落”而非“集合”。

`NestedTable` 不只是一个容器，它还提供了对组级别的操作（`first()`、`last()`、`count()` 等）。这些操作可以被表达式捕获或转成窗口 SQL，从而保持性能路径一致。

## 5. 为什么连续分组更“工程化”

连续分组是一个更接近业务语义的抽象：

- 在交易中，一个“上涨段”可能对应一个策略周期
- 在日志中，一个“异常段”对应一次事故窗口
- 在用户行为中，一个“连续活跃段”体现一次真实的参与过程

如果只能用传统 groupby，你需要手动构造断点列，维护复杂的分组逻辑。LTSeq 把这件事固化为系统能力，让序列分析更自然。

## 6. 权衡：连续语义带来的差异

连续分组的价值很明确，但也带来一些需要注意的差异：

- **与传统 groupby 语义不同**：结果可能比预期“多组”，因为每个断点都会生成新组
- **依赖排序状态**：如果顺序不稳定，连续段的意义会被破坏
- **复杂过滤需要 fallback**：复杂的 group 级条件可能走 pandas 路径

这些差异不是缺陷，而是连续语义的必然结果。它要求你在“序列思维”上更精确，但换来的表达力也更强。

## 7. 价值总结：连续段比全局分组更接近真实世界

传统 groupby 更像“统计思维”，连续分组更像“过程思维”。在很多业务里，过程比统计更重要：

- 你关心的是趋势的形成与中断
- 你关心的是状态的持续时间
- 你关心的是序列里那些可被解释的片段

LTSeq 让这些问题变成直接表达，而不是被迫用曲线救国的方式解决。
