# LTSeq 入门：为什么“序列优先”的表能改变分析方式

当你的数据来自事件流、交易流、日志流或时间序列时，传统数据工具会在不知不觉中制造摩擦。它们把“表”默认成一个集合：行与行之间没有顺序，任何一行都可以被独立地抽取、聚合、比较。但真实世界的序列数据恰恰相反——顺序不是附属信息，而是问题本身。

如果你经常需要回答这些问题，你就会感受到这个错位：

- 这段时间里，连续上涨的区间有哪些？
- 上一个状态到下一个状态之间发生了什么？
- 在滚动窗口里，指标如何演化？

这不是语法问题，而是心智模型的问题。LTSeq 的核心就是在这一点上做了明确选择：**把顺序从隐式约束变成显式能力**。它不只是一个更快的 DataFrame，而是一个更适合序列思维的数据引擎。

## 一句话理解 LTSeq

LTSeq 是一个把“表当成序列”来处理的分析引擎。

这句话听起来简单，但它改变了你表达问题的方式。你不再需要用一堆临时列和复杂的 groupby 来模拟“连续”，也不需要在滚动计算中反复拼装逻辑。顺序成为默认语义，很多序列操作就变成了自然表达。

## 一个直观的心智模型

想象一条流水线：每一行数据都是一件经过流水线的物品。你关心的不只是物品本身，还关心它在流水线上的位置、前后关系，以及连续出现的区间。

在这个模型下，LTSeq 的能力变得很直观：

- **窗口函数**：在流水线上开一扇滑动的观察窗。
- **连续分组**：把连续出现的相同状态段标出来。
- **关联引用**：先建立连接，再按需取值，不急着物化整表。

这不是工具层面的“功能列表”，而是一种表达方式的变化。它让你以序列的视角思考，而不是在集合模型里做补丁式拼接。

## 能力地图：LTSeq 解决的三类问题

为了让你快速抓住它的价值，这里用三类能力概括 LTSeq 的核心用途。

### 1) 序列上的窗口计算
滚动统计、shift、diff、累计变化，本质上都是序列关系。LTSeq 把这类能力视为核心，不需要你绕过模型才能实现。

### 2) 连续区间的语义化表达
传统 groupby 只关心“同值聚合”，而连续分组关心的是“同值区间”。LTSeq 的 `group_ordered` 和 `group_sorted` 为这种连续语义提供了直接的表达方式。

### 3) 惰性关联：先连线，后取值
当你只需要部分关联字段时，传统 join 过于沉重。LTSeq 提供了类似 `LinkedTable` 的机制：先建立引用关系，只有在访问关联列时才物化。

## 实现方式（轻量版）

从实现角度，LTSeq 可以理解为“Python 表达 + Rust 执行”的组合：

- **Python 端**负责表达：你写的 lambda 被捕获成表达式树，而不是直接执行。
- **Rust 内核**负责执行：表达式被转成可执行计划，走矢量化与批处理路径。

你会看到一些关键类名，如 `SchemaProxy`、`Expr`、`LinkedTable`、`NestedTable`。它们的意义不是让你记住细节，而是说明 LTSeq 把“表达”和“执行”清晰分层，从而获得可优化与可扩展的空间。

## 工程权衡：强语义带来的代价

LTSeq 的选择也有代价：

- **显式排序的要求**：序列语义依赖顺序，因此很多操作需要你先排序。
- **表达式可序列化限制**：lambda 需要能被捕获成表达式树，复杂 Python 逻辑可能不被支持。
- **fallback 的性能波动**：超出能力边界时可能退回 pandas 路径。

这些权衡并不意外，因为“顺序成为能力”就意味着“顺序必须被严格对待”。如果你的问题本质是序列，这些代价通常是可以接受的。
