# Fallback 策略：当快路径失败时发生什么

任何系统只要足够复杂，就必须回答一个问题：**当最优路径走不通时，系统如何保证正确性？** LTSeq 的答案是 fallback。它在性能路径之外保留了一条“保底路线”，通常是 pandas。这不是妥协，而是一种工程上的弹性设计。

这一篇聚焦 LTSeq 的 fallback 策略：哪些场景会触发 fallback，它如何在正确性与性能之间做权衡，以及为什么这对序列分析尤其重要。

## 1. 痛点：没有 fallback 的系统不够稳

在数据系统里，最糟糕的情况不是“慢”，而是“错”。如果一个表达式超出了引擎能力边界，而系统仍试图用不完整的路径执行，就会造成不可解释的结果。

LTSeq 的选择是：**当能力边界触发时，宁可慢，也要正确**。

## 2. 心智模型：主引擎 + 应急发电机

可以把 LTSeq 的执行体系理解成两层：

- **主引擎**：Rust 内核 + 矢量化快路径
- **应急发电机**：pandas fallback

主引擎负责高性能执行，但当表达式无法被捕获或优化时，系统就切换到 fallback，保证结果正确。

## 3. 触发 fallback 的典型场景

LTSeq 中常见的 fallback 场景包括：

- **表达式无法序列化**：lambda 中包含复杂 Python 逻辑
- **分组过滤/派生过于复杂**：超出 SQL/window 解析能力
- **某些 join 或搜索操作失败**：Rust 内核无法处理特定模式

例如在 `NestedTable.filter()` 中，系统会先尝试 SQL 路径；如果失败，就退回 pandas 逐组计算（见 `py-ltseq/ltseq/grouping/nested_table.py`）。

## 4. fallback 的工程实现路径

LTSeq 的 fallback 往往遵循一个统一模式：

1) 尝试快路径（SQL / Rust）
2) 捕获异常或解析失败
3) 切换到 pandas
4) 重新构建 LTSeq 返回结果

这种“先快后稳”的策略，让系统既能发挥性能优势，也能在复杂场景下保持正确。

## 5. 为什么 fallback 对序列分析特别重要

序列分析常常涉及复杂逻辑：

- 多层条件分支
- 复杂的组级判断
- 业务特定的状态机规则

这些逻辑很难完全被表达式捕获，也很难保证优化路径稳定覆盖。fallback 的存在，让 LTSeq 能在“复杂业务逻辑”面前保持可用性，而不是直接拒绝执行。

## 6. 权衡：稳定性 vs 性能波动

fallback 的优势很明显，但代价也必须承认：

- **性能不稳定**：pandas 路径可能比 Rust 慢很多
- **内存压力增加**：pandas 需要更多内存
- **行为差异风险**：不同路径可能引入细微差异

LTSeq 选择的是“正确性优先”。在复杂逻辑里，稳定结果比速度更重要。

## 7. 如何管理 fallback 的风险

作为使用者，你可以通过以下方式降低 fallback 的不可控性：

- 尽量使用可序列化的表达式
- 明确排序和分组的语义
- 将复杂逻辑拆分为可表达的步骤

这样可以让更多操作走上快路径，减少性能波动。
