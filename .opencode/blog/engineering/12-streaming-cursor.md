# Streaming 与 Cursor：面向大数据的迭代模型

当数据规模增长到“无法整体装进内存”时，很多数据工具就开始变得脆弱。你会看到内存峰值、不可预测的延迟、甚至整个进程被杀掉。LTSeq 的 `Cursor` 与 `scan()` 针对的正是这个问题：**在不物化全表的情况下，仍然保持稳定的数据处理能力**。

这一篇聚焦 LTSeq 的流式迭代模型：它如何利用 Arrow RecordBatch，在 Python 端提供一种可控的“批次式处理方式”。

## 1. 痛点：全量物化在大数据场景下不可承受

传统读取方式（如 `read_csv()`）会一次性加载数据并构建表结构。这在中小规模数据中很方便，但在大文件上会出现三类问题：

- **内存峰值不可控**：加载即膨胀
- **启动延迟巨大**：必须等全量加载完成
- **处理粒度粗**：无法边读边处理

LTSeq 的 `scan()` 是一种针对超大数据集的“工程降维”：它把“全量加载”改成“批次流式”。

## 2. 心智模型：批次流水线

想象你在处理一条很长的流水线：你不需要等所有物品都到达终点才开始加工。你可以每次拿一段，处理完再拿下一段。

这正是 `Cursor` 的语义：

- 每次返回一个批次（RecordBatch）
- 迭代消费，逐批处理
- 只有你调用 `to_pandas()` 时才会全量物化

## 3. 源码机制：Cursor 的核心设计

`Cursor` 定义在 `py-ltseq/ltseq/cursor.py`。它的核心机制是：

1) `scan()` 返回 Rust 内核的 `LTSeqCursor`
2) Python 的 `Cursor` 包装该对象
3) `__next__()` 从 Rust 获取下一批次数据
4) 用 Arrow IPC 反序列化成 `RecordBatch`

这条路径让 Python 端保持轻量，同时避免了逐行的数据搬运。

## 4. Arrow 的角色：数据传输与批处理的桥梁

Cursor 之所以能高效流式，是因为它直接使用 Arrow 的 `RecordBatch`：

- **列式批处理**：减少 Python 层开销
- **零拷贝特性**：降低数据转换成本
- **标准化协议**：与 pandas 或其他工具兼容

换句话说，Arrow 是 LTSeq 与 Python 世界之间的一条高速通道。

## 5. 什么时候用 scan，什么时候用 read_csv

一个简单的原则：

- 如果数据规模适中且需要多次随机访问，用 `read_csv()`
- 如果数据规模巨大或只需要线性处理，用 `scan()`

`scan()` 更像是“流式读入”，适合批处理、ETL、分阶段计算。

## 6. 工程权衡：流式的代价

流式并不是免费午餐，它带来一些明确限制：

- **一次性消费**：Cursor 是可消耗的，遍历一次就结束
- **操作受限**：随机访问、全局排序等操作无法直接完成
- **调试复杂度**：需要理解批次边界

LTSeq 把这些限制显式呈现出来，是为了让你在大数据场景下拥有可控的性能与内存。

## 7. 为什么这对序列分析重要

很多序列场景天生就是“流”的：

- 日志和监控数据不断增长
- 交易数据以流式方式进入
- 设备事件是持续生成的

如果你总是依赖全量物化，你迟早会触碰系统边界。LTSeq 的 Cursor 模型让序列分析更接近真实的数据流动方式。
