# LTSeq 为什么存在：当你的数据是“序列”，而不是“集合”

如果你处理的是事件流、交易流、日志流或时间序列，你会发现一个隐形阻力：绝大多数数据工具默认把“表”当成一个集合。集合的前提是“行之间没有顺序”，但序列数据的核心恰恰是“位置有意义”。当你把序列问题塞进集合模型里，表达就变得笨重，语义也容易走样。

LTSeq 的出现并不是为了再造一个 DataFrame，而是为了解决这类“序列语义缺失”的工程痛点。它从产品层面做了一个很明确的决定：**把顺序从隐式约束变成显式能力**。这会让很多你以为“只能用复杂代码绕过”的问题，直接变成一条清晰的表达。

下面这篇文章更偏产品理解，帮助你先抓住“它解决了什么问题”，而不是马上深入源码细节。

## 1. 典型痛点：序列问题被迫伪装成集合问题

在传统表工具里，你经常会遇到这些不自然的表达：

- **连续区间难以表达**：想找“连续上涨的区间”，你不得不先构造一堆临时列，再用 groupby 进行间接分段。
- **窗口语义不直观**：rolling/shift 明明是序列行为，却被写成复杂的窗口表达式或多次 join。
- **状态变化难以追踪**：需要“前后状态对比”时，常常要手动引入 lag 列、再做自连接。

这些问题不是你不熟练，而是工具的默认心智模型和数据的真实形态不一致。序列数据的价值来自“前后关系”和“连续段”，而集合模型只擅长“聚合与统计”。

LTSeq 的核心价值，是让你**以序列的方式思考和表达**，而不是绕过模型去拼装结果。

## 2. 心智模型：把表想成一条有方向的队列

为了理解 LTSeq，不妨先换一个直观模型：

**把表想成一条流水线。**

- 每一行都是流水线上经过的一件“物品”。
- 行的顺序不是附属信息，而是物品经过的时间/位置。
- 你关心的是“前一件”和“后一件”，以及“某一段连续区间”。

在这个模型里，很多概念变得自然：

- “窗口函数”就是在流水线上开一扇滑动的观察窗。
- “连续分组”就是把同类物品在流水线上的连续段标记出来。
- “关联”并不一定要把两条流水线合并成一条，而是先建立连接，按需取值。

这不是语法层面的变化，而是思维方式的改变。LTSeq 让你站在序列的视角上工作，而不是在集合模型里进行繁琐的补丁式操作。

## 3. LTSeq 的三类核心问题解法

从产品角度看，LTSeq 重点解决三类问题：

### 3.1 序列上的窗口计算
传统工具里，窗口函数往往是“高级功能”，在工程中也容易写得难以维护。LTSeq 把窗口计算当成序列的天然操作：

- 计算上一行、下一行的关联值
- 计算滚动区间的统计量
- 计算随着时间推进的累计变化

因为 LTSeq 的模型承认顺序，这类操作不再是“额外的语法负担”，而是核心能力。

### 3.2 连续区间的语义化表达
很多业务问题需要关注“连续段”，而不是全局分组。例如：

- 用户连续多日活跃的区间
- 价格连续上涨/下跌的段落
- 系统状态连续异常的时间段

LTSeq 的 `group_ordered`、`group_sorted` 不是传统 groupby，而是对“连续性”本身的表达。它让“连续段”的概念成为一等能力，而不是被迫用 groupby + 临时列来模拟。

### 3.3 惰性关联：先连线，再取值
在常规表工具里，关联（join）是重量级动作：它会立刻生成一张大表，把所有字段摊平。很多时候你并不需要这么早做这件事，你只是希望在需要的时候“能拿到关联字段”。

LTSeq 提供了类似 `LinkedTable` 的语义：

- 先建立“引用关系”
- 需要关联列时再物化
- 对源表过滤时避免提前 join 的开销

这对大型维表、低频关联字段非常关键。

## 4. 它不是“更快的 DataFrame”，而是“不同的工程模型”

很多人会把 LTSeq 当成“一个更快的表工具”，但如果只这样理解，就会误判它的价值。LTSeq 的关键不是速度，而是**工程模型的选择**：

- 你不再需要“把序列问题伪装成集合问题”
- 你可以把顺序视为能力，而不是依赖习惯来保持顺序
- 你可以在表达层面对连续段、窗口、关联等序列语义负责

这种模型上的差异，会反过来影响你的数据表达方式，也会让代码更贴近真实业务语义。

## 5. 一个极简的实现视角（不深入源码）

从实现路径上看，LTSeq 采用的是“Python 表达 + Rust 执行”的结构：

- **Python 端**负责表达：你写的 lambda 被捕获成表达式树，而不是直接执行
- **Rust 内核**负责执行：表达式被转成可执行计划，走矢量化路径

在源码里你会看到一些关键概念：

- `SchemaProxy` / `Expr`：把你写的 `r.col` 变成可序列化的表达式
- `_inner`：Rust 内核的入口边界
- `LinkedTable` / `NestedTable`：分别对应惰性关联与连续分组

这不是为了让你记住类名，而是让你意识到：LTSeq 的表达层和执行层是明确分离的，它更像一个“轻量 DSL + Rust 内核”的组合，而不是纯粹的 Python 计算。

## 6. 工程权衡：强语义带来的使用门槛

任何架构选择都有代价。LTSeq 的代价主要是三点：

- **显式排序的要求**：因为顺序是能力，很多操作会要求你先 sort，这对习惯“隐式排序”的用户是门槛。
- **表达式可序列化限制**：你写的 lambda 需要能被捕获成表达式树，意味着复杂的 Python 逻辑不一定能直接支持。
- **fallback 的不可控性**：当表达式超出能力边界时，会退回到 pandas 路径，性能可能波动。

这些权衡是 LTSeq 选择“强语义”的代价，但如果你的问题本质是序列的，它们往往是值得的。

## 7. 为什么值得关注：典型场景

当你面对以下场景时，LTSeq 的模型优势会显得非常明显：

- 交易序列、行情序列、订单生命周期
- 设备日志、服务运行事件流
- 用户行为序列、连续活跃与流失
- 任何需要“区间”和“前后关系”的分析

你会发现，在这些场景下，LTSeq 提供的不是“更快的速度”，而是**更少的表达扭曲**。
